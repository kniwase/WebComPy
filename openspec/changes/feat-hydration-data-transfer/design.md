## Context

WebComPy renders components on the server (SSG/SSR) and hydrates them in the browser. Currently, the browser re-executes all component setups from scratch, re-fetching data that was already resolved on the server. This is a significant performance problem because:

1. PyScript initialization is already slow (~2-5 seconds), and duplicate fetches add more latency.
2. Users see loading states for data that was already available in the server-rendered HTML.
3. Signal values reset to defaults before reactive updates propagate, causing visible content flicker.

This change adds a data transfer mechanism that serializes server-side state into the HTML and restores it during browser-side hydration.

## Goals / Non-Goals

**Goals:**
- Serialize resolved `AsyncResult` states into the HTML payload after SSR rendering
- Serialize `FetchPort` response caches into the HTML payload
- Restore `AsyncResult` states during browser hydration, skipping re-execution
- Restore `FetchPort` response cache during browser hydration, skipping duplicate network requests
- Include a version field in the payload for forward compatibility
- HTML-escape and JSON-validate the payload to prevent XSS
- Only include data that was actually fetched/resolved during SSR (not all possible data)

**Non-Goals:**
- Restoring Signal values (non-AsyncResult) — this is deferred to a follow-up change
- Streaming SSR (data transfer is a single payload)
- Client-to-server data transfer
- Payload compression
- Custom transfer keys
- Persisting data across sessions (localStorage, etc.)
- Changing `useAsyncResult` API

## Decisions

### Decision: Payload format is JSON in a `<script type="application/json">` tag

The transfer payload is a JSON object embedded in a `<script type="application/json" id="__webcompy_data__">` tag at the end of the `<body>`, before the PyScript bootstrap `<script>` tag.

```json
{
  "__webcompy_transfer_version__": 1,
  "fetches": {
    "/api/data": {"status_code": 200, "headers": {...}, "body": "..."}
  },
  "async_results": {
    "component-id-1": {"state": "success", "data": ...}
  }
}
```

**Rationale:** `<script type="application/json">` is not executed by browsers, is easy to parse, and is the standard approach used by Next.js (`__NEXT_DATA__`), Nuxt 3 (`__NUXT_DATA__`), and other frameworks. JSON is safe to embed after HTML-escaping.

**Alternatives considered:**
- Inline `<script>` with a JS variable assignment: Works but requires careful escaping and is less idiomatic.
- Custom element with JSON attribute: Harder to parse, not standard.
- Separate JSON file fetched at runtime: Adds a network round-trip, defeating the purpose.

### Decision: Component IDs as keys for `async_results`

Each `AsyncResult` that was resolved during SSR is keyed by its component's tree path ID (`component._tree_path_id`). This provides a stable, deterministic identifier derived from the component's position in the render tree. Note: `_tree_path_id` is an instance attribute (tree-position path) and is distinct from `ComponentProperty["component_id"]` (registration name, TypedDict field).

**Rationale:** Component IDs are already generated by the framework and are unique per component instance. They provide a natural mapping between server-resolved state and browser component instances.

**Alternatives considered:**
- Using the async function's qualified name: Not unique if the same function is used in multiple components.
- Using array indices: Fragile if component tree changes between SSR and hydration.
- Using a hash of the component path: More complex and not obviously better.

### Decision: `ServerFetchPort` caches all self-site responses for transfer

`ServerFetchPort` maintains an internal response cache keyed by URL. After SSR rendering completes, the cache contents are collected and included in the transfer payload under `fetches`. `BrowserFetchPort` reads the payload and populates its own cache, returning cached responses for matching URLs without making network requests.

**Rationale:** This is the most natural deduplication point. `FetchPort` is the abstraction for HTTP requests; caching at this level means all consumers (including `HttpClient`, `useAsyncResult`, and direct `fetch_port.fetch()` calls) benefit automatically.

**Alternatives considered:**
- Caching at `AsyncResult` level only: Would miss direct `FetchPort.fetch()` calls that don't go through `useAsyncResult`.
- Caching at `HttpClient` level: Too high-level; `HttpClient` is browser-only.

### Decision: `AsyncResult._restore_from_transfer()` bypasses async execution

When a transfer payload entry matches an `AsyncResult`'s component ID, the `_restore_from_transfer()` method sets `_state` directly to `SUCCESS`, sets `_data` to the transferred value, and skips `_execute()`. This avoids the `LOADING` state entirely for transferred data.

**Rationale:** The data was already resolved on the server; re-executing the async function would be wasteful and would show a loading flash. Direct state restoration is the most efficient path.

**Alternatives considered:**
- Setting `_state` to `LOADING` then transitioning to `SUCCESS`: Causes a flash of loading state.
- Re-executing the async function but with a pre-warmed cache: Still slower than direct restoration and more complex.

### Decision: Version field for forward compatibility

The payload includes `__webcompy_transfer_version__: 1`. If the payload format changes in the future, the version number will be incremented, and the browser-side deserializer can handle multiple versions.

**Rationale:** This is a standard practice for data formats that may evolve. It costs almost nothing (one integer in the JSON) and prevents breaking changes.

### Decision: Payload injected by `generate_html()` and read during `app.run()`

The server side injects the payload in `generate_html()` (after rendering completes and transfer data is collected). The browser side reads the payload during `app.run()` initialization (before component setup begins).

**Rationale:** `generate_html()` is the single point where HTML output is assembled. `app.run()` is the single entry point for browser initialization. This placement ensures the payload is available before any component setups execute.

### Decision: Only include resolved `AsyncResult` states (not PENDING/LOADING/ERROR)

The `async_results` section only includes entries where `state` is `SUCCESS`. `PENDING`, `LOADING`, and `ERROR` states are not transferred. `ERROR` states could be transferred in the future, but for the initial implementation, only successful data is transferred.

**Rationale:** Transferring `LOADING` or `PENDING` states is meaningless (no data to restore). Transferring `ERROR` states is useful but adds complexity (error types, messages, stack traces) that can be addressed in a follow-up change.

## Risks / Trade-offs

- **[Risk] Payload size** — For applications that fetch large amounts of data, the payload could significantly increase HTML size. → **Mitigation:** Only resolved data is included; PENDING/LOADING/ERROR states are excluded. Developers can control payload size by not fetching excessive data during SSR. Compression can be added in a future change.

- **[Risk] Stale data** — If the server caches are warmed from a previous request and the data has changed since, the transferred data may be stale. → **Mitigation:** This is acceptable for SSG (static sites serve fixed content). For dynamic SSR, the data is always fresh from the current render. No caching across requests is involved.

- **[Risk] XSS via payload injection** — A malicious actor could inject content into the payload. → **Mitigation:** The payload is JSON-serialized and HTML-escaped before embedding. The browser-side deserializer validates the JSON structure and version field before processing.

- **[Risk] Component ID stability** — If component IDs change between SSR and browser hydration, `AsyncResult` states won't be restored. → **Mitigation:** Component IDs are generated deterministically from the component tree structure. As long as the same component tree is rendered (which hydration guarantees), IDs match.

- **[Risk] Cross-environment serialization** — Python objects serialized on the server must be deserializable in the browser (PyScript/Pyodide). → **Mitigation:** The payload uses JSON, which is universally supported. Custom objects are not supported — only JSON-serializable types (str, int, float, bool, None, list, dict) are transferred. Non-serializable data is silently excluded with a warning.

## Implementation Outline

1. Create `webcompy/hydration/` module with payload schema, serializer, and deserializer.
2. Add response caching to `ServerFetchPort` and cache collection method.
3. Add `_restore_from_transfer()` to `AsyncResult`.
4. Add transfer data collection to `AppDocumentRoot` / `WebComPyApp` after SSR rendering.
5. Inject payload `<script>` tag in `generate_html()`.
6. Read payload in `app.run()` and restore `AsyncResult` states and `BrowserFetchPort` cache.
7. Add unit tests for serialization, deserialization, state restoration, and cache population.
8. Add E2E tests for full SSR→browser data transfer flow.

## Open Questions

1. Should `ERROR` state `AsyncResult` entries be transferred? (Current decision: No, only `SUCCESS` states are transferred. Can be added later.)
2. Should the payload be placed in `<head>` or at the end of `<body>`? (Current decision: End of `<body>`, before the PyScript bootstrap `<script>`, so it doesn't block initial HTML parsing.)
3. **Future: Signal value restoration** — Signal values (non-AsyncResult) are NOT transferred in this change. Only `AsyncResult` states and `FetchPort` responses are transferred. Signal value restoration can be added as a follow-up if flash-of-default-content is a significant issue.
