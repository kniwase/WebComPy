## Context

WebComPy applications mount inside a `<div id="webcompy-app">` element. The surrounding `<html>` tag is generated by the SSG pipeline (`generate_html()`) and is not part of the reactive component tree. Currently, `AppDocumentRoot` manages `<head>` content through `HeadPropsStore`, but has no mechanism to influence the `<html>` element itself. This limits developers who need to set `class`, `lang`, or `data-*` attributes on the root element — a common requirement for CSS frameworks like Tailwind that use class-based dark mode.

## Goals / Non-Goals

**Goals:**
- Allow developers to set static and reactive attributes on the `<html>` element from Python code
- Support both browser (live DOM) and SSG (pre-rendered HTML) environments
- Maintain per-app isolation (each `WebComPyApp` manages its own `<html>` attributes)
- Keep the API consistent with existing `set_head` / `append_script` patterns

**Non-Goals:**
- Managing `<body>` or `<head>` attributes (already handled by existing systems)
- Supporting nested `<html>` elements or child manipulation
- Reactive attribute observation from external scripts

## Decisions

### Decision: Store attributes on `AppDocumentRoot`

`AppDocumentRoot` already owns head management and rendering lifecycle. Storing `<html>` attributes here maintains per-app isolation and keeps the API surface consistent.

**Alternatives considered:**
- Store on `WebComPyApp` directly: fragments responsibility across two classes
- Use DI for html attrs: overkill for simple key-value storage
- Pass through `AppConfig`: doesn't support reactive values

### Decision: Accept both `str` and `Computed[str]` values

This matches the existing pattern in `HeadPropsStore` where reactive values are supported for title/meta. For SSG, `Computed.value` is read at render time.

### Decision: Apply browser-side changes in `_render()`

The `_render()` method on `AppDocumentRoot` already has access to the DOM via `browser.document`. Adding a hook here to sync `html_attrs` to the live `<html>` element keeps the timing correct (after initial render, before loading screen removal).

### Decision: Use `_HtmlElement` in `generate_html()` to emit attributes

The `_HtmlElement` helper class already supports arbitrary attributes. Passing `html_attrs` dict into its constructor is the minimal change to the SSG pipeline.

## Risks / Trade-offs

- **[Risk]** SSG output uses `html_attrs` captured at render time; if a `Computed` value changes later, the static HTML won't reflect it → **Mitigation:** Document this limitation; for dynamic attributes, use client-side hydration
- **[Risk]** Multiple apps on one page could conflict over `<html>` attributes → **Mitigation:** Document that only the most recently rendered app's html attrs take effect; this aligns with the existing module-level DI fallback behavior
- **[Risk]** Circular import if `Computed` is imported at module level in `_root_component.py` → **Mitigation:** Use `TYPE_CHECKING` guard or import inside methods

## Migration Plan

No migration needed — this is an additive feature. Existing code continues to work unchanged.
